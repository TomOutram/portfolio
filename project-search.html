Overview

The central website for the Ministry of Education is education.govt.nz. The site recently underwent a refresh, with a content overhaul and visual redesign.

I worked as a UX/UI designer on this project. One of my key tasks was to design the user interface for the site's global search which was built using Algolia.

This required clear, readable search results and easy to use filtering options.

Challenges

Categories and tagging - The site's taxonomy included many different variables such as; audiences, page types, document types, media types, and more. It was difficult to keep everything tidy and understandable for users.

Technical limitations - My designs had to work within the constraints of Algolia's out-of-the-box features. I worked closely with the developers to make sure what I designed was feasilbe and realistic.

Limited site content - The development of the search functionality didn't align with the content re-writing. It was hard to test a site search without content to search for.

Information architecture

IA in figjam

The site redesign focused on organising content around the audience. Content is tagged by audience type and automatically placed in the right area, with filters in the global search that allow users to search by audience.

The site has three levels: Level 1 is the audience (e.g., ‘Parents and caregivers’), Level 2 is a sub-group (e.g., ‘early childhood education’), and Level 3 is a specific topic (e.g., ‘Choosing a kindergarten’). My goal for the search results was to make it clear where each page fits within this structure, helping users easily find relevant content.

Taxonomy & tagging

tagging concepts and page types

Outside of audience types, I needed to break down several other entities within the site’s taxonomy into searchable terms. These content types would dictate how everything loaded onto the site was tagged, making it searchable and filterable by users.

I wanted to make sure the tags were clear and readable. Users should be able to scan a list of search results and quickly recognise each type. These included:

  • Page type - Pages such as; general information, news, key dates, or consultations and surveys.
  • Compliance type - Some of the site contains guidance that educators must comply with. Other content is only recommended or simply general information.
  • Media type - This includes media featured on the site such as; PDF documents, diagrams, graphs, videos, or audio files.

Search results

Document search page design concept

When designing the search results UI, I aimed to clearly convey the audience and content types I had established. I went through several design iterations to balance providing enough context without overwhelming users with too much information. Key considerations included:

  • Scannability - I made the titles the most prominent element of the component so users could visually bounce between them as they scanned the results. I used thumbnails and icons to help make the document and media results visually distinct.
  • Meta data - I iterated over different variations to create a balance between the amount of info displayed and the readability of the components. I didn't want to overwhelm users with details.
  • Interaction - Users should intuitively know where to click to select a result. I used our design system's 'interactive' blue for headings and downloads with a colour change and underline on-hover to indicate interactivity.
  • Navigation - I included a breadcrumb pathway to help users understand the location of the page in relation to parent pages in the site's hierarchy of information.

Search filters

filters design iterations

I went through several versions of the global site search filters, trying to determine the best order and which filters were most important for users. This was challenging due to the many audience types. I also debated whether dropdowns or checkboxes were better for certain filters, keeping in mind the need to future-proof for new content types.

Eventually, I developed a solid first version that worked well for all content types. The next step was usability testing to see if it made sense to users.

Usability testing

Example questions from the testing plan
Analysing test results on sticky notes in Figjam

We ran several rounds of usability testing with users. Initially, I used high-fidelity mockups of the search and filters to gather feedback. Based on user input, we made changes like simplifying the 'type' tags, renaming some filters, and rearranging the order of the filters.

Based on these changes, our developers implemented the search designs into our UAT environment for more in-depth testing with users. I created a guide with questions we could follow during the testing sessions. We prompted users to perform various tasks so we could observe their behaviour and interactions with the search components.

I collated and analysed the findings on a Figjam board. The most actionable data was organised into groups and prioritised in our Jira backlog.

Test feedback

Hierarchy filter design iterations
Final heirarcy filter designs

User testing revealed technical bugs, simple usability improvements, and a few larger design changes we needed to make.

A key change was making the ‘search in’ option more prominent and user-friendly. I went through several design iterations, testing them with users to ensure it stood out and was easy to understand. The final design was built using a customised version of algolia’s hierarchy filter, allowing users to dig down to the specific type of content.

Summary

Final design for results component
Final design of the site search results page

Outcomes - The new filter designs were added to the site, and feedback was mostly positive, though there's still room for improvement. The search works better for some users than other depending on what they’re looking for.

What did I learn? - This project taught me a lot about designing a large, content-heavy website on a CMS.

  • I learned about site searches and the technology we used to build ours (Algolia).
  • I had the chance to test my designs multiple times, analysing them from different angles and using feedback to improve each version. This taught me how to gather actionable insights from users and apply them to my design work.
  • I learned the value of simplicity. Our final design is much more streamlined than the original, but I believe it could be simplified even further.

Next steps - Going forward, we'll continue adjusting settings and reorganising filters to find the best balance for all audiences.
